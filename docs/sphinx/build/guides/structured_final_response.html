Structured Final Response Pattern

Overview

The Structured Final Response pattern is a fundamental architectural approach for agent-to-application integration. It solves the problem of passing machine-readable, actionable data from the agent’s reasoning loop to external systems (like React frontends) without fragile string parsing.

The Problem

Previously, agents returned simple text responses:

"I have successfully created the 'Total Profit' measure in the 'sales' table."

This is great for chat logs but:

  • ❌ Not machine-readable

  • ❌ Requires brittle string parsing

  • ❌ No clear contract between agent and frontend

  • ❌ Difficult to trigger specific UI behaviors

The Solution

Agents now return structured FinalResponse objects with:

  • operation: High-level command for the frontend (e.g., “display_table”, “model_ops”)

  • payload: JSON data to execute the operation

  • human_readable_summary: Natural language for chat display

FinalResponse(
    operation="model_ops",
    payload={
        "proposed_ops": [
            {
                "action": "addRelationship",
                "relationship": {
                    "from": {"table": "LU_EMPLOYEE", "column": "EmployeeKey"},
                    "to": {"table": "FACT_SALES", "column": "EmployeeKey"},
                    "isActive": true,
                    "fromCardinality": "one"
                }
            }
        ]
    },
    human_readable_summary="Proposed adding relationship between LU_EMPLOYEE and FACT_SALES."
)

Architecture Changes

1. Core Type (framework/base.py)

class FinalResponse(BaseModel):
    """Pydantic model for structured agent responses"""
    operation: str = Field(..., description="Frontend operation type")
    payload: Dict[str, Any] = Field(..., description="Operation data")
    human_readable_summary: str = Field(..., description="Chat display text")

2. Agent Loop (framework/core/agent.py)

async def run(self, task: str) -> FinalResponse:
    # ... planning loop ...
    
    if isinstance(plan_outcome, FinalResponse):
        # Publish full structured response
        end_data = {
            "result": plan_outcome.model_dump(),
            "operation": plan_outcome.operation,
            "payload": plan_outcome.payload,
            "summary": plan_outcome.human_readable_summary
        }
        self.event_bus.publish("agent_end", end_data)
        return plan_outcome  # Return structured object

3. Planner Parsing (framework/components/planners.py)

The ReActPlanner now parses both:

  • New format: {"final_response": {...}}

  • Legacy format: {"final_answer": "..."}

# Check for new structured final_response format
if "final_response" in decision:
    final_resp_data = decision["final_response"]
    return FinalResponse(
        operation=final_resp_data.get("operation", "display_message"),
        payload=final_resp_data.get("payload", {}),
        human_readable_summary=final_resp_data.get("human_readable_summary", "Task completed.")
    )

# Legacy support: simple final_answer string
if decision.get("final_answer"):
    return FinalResponse(
        operation="display_message",
        payload={"message": decision["final_answer"]},
        human_readable_summary=decision["final_answer"]
    )

4. LLM Prompt Instructions (configs/agents/pbi/schema_editor_function_calling.yaml)

The agent prompt now explicitly instructs the LLM on how to format final responses:

system_prompt: |
  ...
  
  FORMAT - Return JSON with a "final_response" key:
  {
    "thought": "I have completed the task...",
    "final_response": {
      "operation": "<operation_type>",
      "payload": { ...data... },
      "human_readable_summary": "Natural language summary"
    }
  }

Supported Operations

1. display_message

Simple text responses.

{
  "operation": "display_message",
  "payload": {"message": "Here are the results..."},
  "human_readable_summary": "Here are the results..."
}

2. display_table

Structured tabular data.

{
  "operation": "display_table",
  "payload": {
    "title": "Model Tables",
    "headers": ["Table Name", "Row Count"],
    "rows": [["sales", "1000"], ["products", "500"]]
  },
  "human_readable_summary": "Found 2 tables in the model."
}

3. model_ops

Proposed Power BI model operations.

{
  "operation": "model_ops",
  "payload": {
    "proposed_ops": [
      {
        "action": "addRelationship",
        "relationship": {
          "from": {"table": "LU_EMPLOYEE", "column": "EmployeeKey"},
          "to": {"table": "FACT_SALES", "column": "EmployeeKey"},
          "isActive": true,
          "fromCardinality": "one"
        }
      }
    ]
  },
  "human_readable_summary": "Proposed relationship between tables."
}

4. update_file_tree

File system modifications.

{
  "operation": "update_file_tree",
  "payload": {
    "changed_files": ["Model/tables/sales.tmdl"],
    "details": "Added DAX measure 'Total Profit'"
  },
  "human_readable_summary": "Successfully added measure."
}

Frontend Integration

WebSocket Handler

# FastAPI WebSocket endpoint
final_result = await agent.run(task)

await websocket.send_json({
    "event": "agent_end",
    "data": {
        "result": final_result.model_dump(),
        "operation": final_result.operation,
        "payload": final_result.payload,
        "summary": final_result.human_readable_summary
    }
})

React Frontend

// React component handling agent responses
const handleAgentEnd = (data: AgentEndData) => {
  switch (data.operation) {
    case "display_table":
      return <DataTable {...data.payload} />;
    
    case "model_ops":
      return <ModelOperationsList ops={data.payload.proposed_ops} />;
    
    case "update_file_tree":
      refreshFileTree(data.payload.changed_files);
      return <ChatMessage text={data.summary} />;
    
    case "display_message":
    default:
      return <ChatMessage text={data.summary} />;
  }
};

Why This Approach is Superior

1. Decoupling

The agent doesn’t need to know about React or frontend details. It just produces a structured payload in an agreed-upon format.

2. Type Safety

Pydantic validation ensures the contract is enforced. No runtime surprises.

3. Extensibility

Need a new UI interaction? Just:

  • Define a new operation (e.g., "display_chart")

  • Add it to the prompt documentation

  • Implement the corresponding React component

No framework changes needed!

4. Clear Separation of Concerns

  • Agent: Responsible for generating the final result

  • Application: Responsible for rendering the final result

5. Backward Compatibility

The system supports both new structured responses and legacy string responses, ensuring smooth migration.

Decision Tree: When to Use What

Task Completed?
├─ Yes
│  ├─ Need special UI rendering? (table, file tree, ops preview)
│  │  └─ Use final_response with appropriate operation
│  │
│  └─ Simple conversational response?
│     └─ Use complete_task OR final_response with "display_message"
│
└─ No
   └─ Return Action(s) for next step

Migration Guide

For Existing Agents

  1. No immediate changes required - Legacy complete_task still works

  2. Optional enhancement - Add final_response instructions to prompts

  3. Gradual adoption - Start with one operation type, expand over time

For Frontend Developers

  1. Listen for agent_end event

  2. Check data.operation field

  3. Implement handlers for each operation type

  4. Fall back to data.summary for unknown operations

Best Practices

  1. Use descriptive operation names - Make it obvious what the frontend should do

  2. Include rich metadata in payload - Give frontend all the context it needs

  3. Always provide human_readable_summary - Fallback for chat/logging

  4. Validate payloads - Use TypeScript interfaces or JSON Schema on frontend

  5. Document new operations - Update this file when adding new operation types

Summary

✅ Type-safe structured outputs with Pydantic
✅ Decoupled agent and frontend architectures
✅ Extensible - add new operations without framework changes
✅ Backward compatible - supports legacy responses
✅ Clear contract between agent and application layers

This pattern transforms agent outputs from “strings to parse” into “commands to execute”, making the entire system more reliable, maintainable, and extensible.
